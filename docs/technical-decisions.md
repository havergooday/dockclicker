# 기술 결정 기록

## 목적

이 문서는 프로젝트에서 이미 내려진 기술적 판단과 그 이유를 기록한다.  
AI나 사람이 나중에 다시 작업할 때 같은 결정을 반복해서 고민하지 않도록 하는 것이 목적이다.

## 결정 1: 엔진은 Godot 4.6을 사용한다

- 상태: 결정됨
- 이유:
  - Steam PC용 2D 데스크톱 컴패니언 게임에 적합하다
  - Unity급 워크플로우보다 가볍고, 순수 라이브러리보다 생산성이 높다
  - 가로형/세로형 반응형 UI와 SD 캐릭터 연출을 다루기 좋다

## 결정 2: 프로젝트 구조는 `출시형 구조 + MVP 범위`로 시작한다

- 상태: 결정됨
- 이유:
  - 처음부터 확장 가능한 책임 분리를 유지한다
  - 동시에 첫 구현 범위는 작게 가져가 개발 속도를 확보한다

## 결정 3: 문서 허브 중심 구조를 유지한다

- 상태: 결정됨
- 이유:
  - 프로젝트를 다시 열었을 때 AI가 빠르게 컨텍스트를 복구할 수 있다
  - 문서가 늘어나도 `start-here` 와 `document-index` 를 통해 진입 경로를 유지할 수 있다

## 결정 4: 구조 검증은 Python unittest로 유지한다

- 상태: 결정됨
- 이유:
  - Godot 에디터 실행 전에도 빠르게 필수 파일과 설정을 검증할 수 있다
  - 문서/구조 변경을 가볍게 자동 확인할 수 있다

## 결정 5: 메인 화면 구조는 `Hangar / Operations / Bridge Lounge`를 기본으로 한다

- 상태: 결정됨
- 이유:
  - 가로형 기준에서 장비, 운영, 인원 정보 흐름이 명확하다
  - SD 캐릭터와 운영 대시보드를 동시에 살리기 좋다

## 결정 6: 파츠는 수량 스택이 아닌 개별 아이템 인스턴스로 관리한다

- 상태: 결정됨
- 이유:
  - 향후 드랍 시스템(고난이도 행성 보상)에서 개별 아이템이 생성되어야 한다.
  - 파츠별 고유 속성(내구도, 특성, 강화 수치 등) 확장을 위한 기반이 필요하다.
  - 수량 카운트 방식은 나중에 인스턴스 방식으로 바꾸면 저장 데이터 마이그레이션이 필요해 조기 결정이 유리하다.
- 영향:
  - `GameState`의 파츠 인벤토리는 `Dictionary(type → count)` 대신 `Array[PartInstance]` 형태로 구성해야 한다.
  - 현재 구현(`owned_parts` 수량 딕셔너리)은 이 모델로 리팩터링이 필요하다.
  - UI는 동일 종류끼리 시각적으로 묶어 표시할 수 있으나 내부 데이터는 인스턴스 배열을 기준으로 한다.

## 아직 미정인 기술 결정

- 저장 포맷: JSON / ConfigFile / Resource 기반 중 무엇으로 갈지
- 세로형 레이아웃 전환 방식: 별도 씬 / 동적 재배치
- SD 캐릭터 애니메이션 방식: 프레임 애니메이션 / AnimationPlayer 중심
- 오프라인 진행 계산 위치: 오토로드 / 별도 서비스 객체

## 갱신 원칙

- 기술 선택이 실제로 확정됐을 때만 이 문서를 갱신한다.
- 단순 아이디어나 미정 상태는 `현재 우선순위`나 관련 설계 문서에 둔다.
